3. ENDPOINTS DISPONIBLES
────────────────────────────────────────────────────────────────────────────────

RÉFÉRENTIELS — lecture publique (aucun token requis)
  GET /niveaux/           → List[NiveauRead]      (pour dropdown frontend)
  GET /niveaux/{id}       → NiveauRead
  GET /series/            → List[SerieRead]
  GET /series/{id}        → SerieRead
  GET /interets/          → List[InteretRead]     (trié par designation)
  GET /interets/{id}      → InteretRead

AUTH
  POST /auth/register     → UserRead              (inscription)
  POST /auth/login        → Token                 (connexion)
  GET  /auth/me           → UserRead + profile imbriqué

PROFIL — token requis
  GET /profiles/me        → ProfileRead           (profil complet avec niveau/serie/interets)
  PUT /profiles/me        → ProfileRead           (créer ou mettre à jour)
    Body : {
      "ville":       "Dakar",
      "bio":         "Étudiant en Terminale S1...",
      "niveau_id":   4,          ← id depuis GET /niveaux/
      "serie_id":    1,          ← id depuis GET /series/
      "interet_ids": [1, 2, 5]   ← ids depuis GET /interets/
    }

CHAT RAG
  POST /chat/             → ChatResponse
    Body : { "message": "...", "session_id": "..." }
    Header : Authorization: Bearer <token>  (optionnel)


────────────────────────────────────────────────────────────────────────────────
4. CYCLE DE VIE D'UNE REQUÊTE POST /chat/
────────────────────────────────────────────────────────────────────────────────

ÉTAPE 1 — Auth optionnelle
  Token présent → SELECT users + selectinload(profile → niveau, serie, interets)
  Pas de token  → current_user = None

ÉTAPE 2 — Chargement/création de la conversation (si authentifié)
  SELECT conversations + selectinload(messages) WHERE session_id AND user_id
  Si non trouvé → INSERT conversation + flush (id généré, pas encore commité)

ÉTAPE 3 — Fenêtre d'historique
  history = [{"role": m.role, "content": m.content} for m in messages[-10:]]
  (5 derniers échanges = 10 messages)

ÉTAPE 4 — Construction du user_context
  {
    prenom:   current_user.prenom,
    nom:      current_user.nom,
    niveau:   profile.niveau.designation  (ou "" si non renseigné),
    serie:    profile.serie.designation   (ou ""),
    interets: [i.designation for i in profile.interets],
  }

ÉTAPE 5 — RAG
  5a. BGE-M3 encode la question → vecteur 1024D
  5b. ChromaDB top-k chunks
  5c. _build_system_prompt(user_context) → prompt personnalisé avec profil
  5d. Ollama : [system + historique + question+chunks] → réponse

ÉTAPE 6 — Persistance (si authentifié)
  INSERT messages x2 (user + assistant)
  UPDATE conversations.updated_at
  db.commit()  ← tout en une seule transaction

ÉTAPE 7 → { answer, sources, session_id }


────────────────────────────────────────────────────────────────────────────────
5. WORKFLOW FRONTEND RECOMMANDÉ
────────────────────────────────────────────────────────────────────────────────

1. Au chargement de l'app (avant inscription) :
   GET /niveaux/ → stocker en cache local (dropdown)
   GET /series/  → stocker en cache local
   GET /interets/→ stocker en cache local (checkboxes)

2. Inscription :
   POST /auth/register  { email, nom, prenom, password }
   → Retourne UserRead sans profil (profile = null)

3. Complétion du profil (onboarding) :
   PUT /profiles/me  { niveau_id, serie_id, interet_ids, ville, bio }
   → Retourne ProfileRead complet

4. Affichage du profil :
   GET /auth/me → UserRead avec profile imbriqué
   (ou GET /profiles/me pour juste le profil)

5. Chat :
   POST /chat/  { message, session_id? }  + Bearer token
   → Kali répond en connaissant le profil (niveau, série, intérêts)
   → Stocker le session_id retourné pour la prochaine requête

6. CE QUI EST STOCKÉ / NON STOCKÉ
────────────────────────────────────────────────────────────────────────────────

STOCKÉ EN BASE :
  ✓ messages.role + content + created_at  (table dédiée, paginable)
  ✓ profiles.ville + bio + niveau_id + serie_id
  ✓ profile_interets (junction ids)
  ✓ niveaux/series/interets (référentiels pré-peuplés)

NON STOCKÉ :
  ✗ Chunks ChromaDB (recalculés à chaque requête)
  ✗ Vecteurs BGE-M3
  ✗ System prompt (reconstruit dynamiquement)
  ✗ Sources PDF par échange (retournées dans la réponse uniquement)
